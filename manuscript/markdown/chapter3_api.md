## 3.1 มาตรฐานการสื่อสารข้อมูลในงานเกษตรอัจฉริยะ
ในอดีต ข้อมูลทางเกษตรกรรมมักถูกจัดเก็บแยกส่วนแบบปิด (Data Silos) บนคอมพิวเตอร์ประจำฟาร์ม ทำให้ไม่สามารถแชร์หรือประมวลผลร่วมกันได้ ในยุคเกษตรกรรมดิจิทัล การเชื่อมต่อข้อมูลอย่างไร้รอยต่อจึงต้องพึ่งพา **"โปรโตคอลการสื่อสารข้อมูล" (Communication Protocols)** ที่เป็นมาตรฐานเปิดสากล เพื่อเป็นกติกาในการแลกเปลี่ยนข้อมูลระหว่างอุปกรณ์ต่างยี่ห้อ

สถาปัตยกรรมโปรโตคอลหลักในงานวิศวกรรมข้อมูลเกษตร แบ่งออกเป็น 2 ค่ายสำคัญตามลักษณะงาน:

```
                  ┌──────────────────────────────────────────────┐
                  │    ระบบสถาปัตยกรรมโปรโตคอลการสื่อสารเกษตร       │
                  └──────────────────────┬───────────────────────┘
                                         │
                 ┌───────────────────────┴───────────────────────┐
                 ▼                                               ▼
         ค่ายส่งข้อมูลอุปกรณ์ภาคสนาม                             ค่ายดึงข้อมูลจากแหล่งคลาวด์
           (MQTT Protocol)                                 (HTTP/HTTPS Protocol)
  - ทำงานแบบ Request-Response                     - ทำงานแบบ Publish-Subscribe
  - ขนาดหัวเฟรมข้อมูลใหญ่ (Overhead สูง)             - ขนาดหัวเฟรมเล็กมาก (1-2 Bytes)
  - เหมาะสำหรับดึงสถิติ/พยากรณ์อากาศ                 - เหมาะสำหรับส่งค่าเซ็นเซอร์ต่อเนื่อง
```

### 1. โปรโตคอลการโอนย้ายข้อความหลายมิติ (Hypertext Transfer Protocol - HTTP/HTTPS)
* **กลไก**: ทำงานบนสถาปัตยกรรมการร้องขอและการตอบกลับ (Request-Response Architecture) โดยอุปกรณ์ปลายทาง (Client) จะต้องส่งคำขอไปยังเซิร์ฟเวอร์ (Server) ก่อน เซิร์ฟเวอร์จึงจะส่งข้อมูลตอบกลับมา
* **การประยุกต์ใช้**: เหมาะสำหรับการดึงข้อมูลพยากรณ์อากาศขนาดใหญ่จากสถานีอุตุนิยมวิทยาสากล หรือการอัปโหลดสรุปผลผลิตรายวันขึ้นไปเก็บบนระบบคลาวด์
* **ข้อจำกัด**: มีขนาดหัวเฟรมข้อมูล (Header Overhead) ที่ใหญ่ จึงสิ้นเปลืองพลังงานและแบนด์วิดท์เครือข่ายสูงหากส่งข้อมูลถี่เกินไป

### 2. โปรโตคอลการส่งข้อความคิวต่อเนื่องน้ำหนักเบา (Message Queuing Telemetry Transport - MQTT)
* **กลไก**: ทำงานบนสถาปัตยกรรมการลงทะเบียนและแพร่กระจายข้อความ (Publish-Subscribe Architecture) โดยมีตัวกลางจัดการข้อความ (Message Broker) ทำหน้าที่รับข้อความจากผู้เผยแพร่ (Publisher เช่น เซ็นเซอร์) และส่งต่อไปยังผู้รับบริการ (Subscriber เช่น ตัวควบคุมวาล์วน้ำ)
* **การประยุกต์ใช้**: เหมาะสำหรับเครือข่ายเซ็นเซอร์ในสวนทุเรียนหรือยางพาราที่ต้องการส่งค่าความชื้นดิน อุณหภูมิ และแสง ทุก ๆ 5-10 วินาที เนื่องจากหัวเฟรมข้อมูลมีขนาดเล็กมาก (เริ่มต้นเพียง 2 ไบต์) ประหยัดแบตเตอรี่ได้สูงสุด

## 3.2 สถาปัตยกรรมข้อมูลเจสันและการจัดการข้อมูลกึ่งโครงสร้าง
ข้อมูลที่ได้จากการตรวจวัดภาคสนามมักมีลักษณะเป็น **"ข้อมูลกึ่งโครงสร้าง" (Semi-structured Data)** เนื่องจากแต่ละสถานีตรวจอากาศอาจติดตั้งชนิดเซ็นเซอร์ไม่เท่ากัน (เช่น สถานีที่ 1 มีเซ็นเซอร์วัดแรงลม แต่สถานีที่ 2 มีเพียงเซ็นเซอร์วัดน้ำฝน) การใช้ฐานข้อมูลแบบตารางดั้งเดิม (Relational Database) จึงมีความแข็งกระด้างและเสียพื้นที่จัดเก็บสูง

มาตรฐานการส่งข้อมูลในรูปแบบ **เจสัน (JavaScript Object Notation - JSON)** จึงได้รับการยอมรับเป็นภาษากลางในการลำเลียงข้อมูลดิจิทัลเกษตร เนื่องจากมีความยืดหยุ่นสูง อ่านง่ายด้วยมนุษย์ และแปลงผลได้รวดเร็วด้วยคอมพิวเตอร์

### โครงสร้างองค์ประกอบภายใน JSON:
1. **คู่คีย์และค่า (Key-Value Pair)**: จับคู่คีย์ที่เป็นอักขระกับค่าชนิดต่าง ๆ เช่น `"leaf_temperature": 32.5`
2. **คลาสอ็อบเจกต์ซ้อนซ้อน (Nested Objects)**: การจัดกลุ่มย่อยพารามิเตอร์เพื่อให้สะท้อนมิติความสัมพันธ์ เช่น
   ```json
   {
       "station_name": "RBRU-Chanthaburi",
       "gps": {"latitude": 12.6, "longitude": 102.1},
       "reading": {"moisture": 24.5}
   }
   ```
3. **รายการข้อมูลแถวลำดับ (Arrays)**: การจัดเก็บชุดข้อมูลลำดับเวลา เช่น `[{"hour": 1, "temp": 28.5}, {"hour": 2, "temp": 29.0}]`

## 3.3 วิศวกรรมการเชื่อมต่อเอพีไอสำหรับสถานีตรวจอากาศ
การบูรณาการระบบนิเวศดิจิทัลจำเป็นต้องพัฒนาทักษะ **"วิศวกรรมเอพีไอ" (API Engineering)** เพื่อทำหน้าที่เป็นสะพานเชื่อมต่อดึงข้อมูลจากแพลตฟอร์มพยากรณ์อากาศภายนอก (เช่น OpenWeather หรือระบบจำลองสภาพอากาศภาครัฐ) เข้าสู่โปรแกรมสนับสนุนการตัดสินใจของเรา

การเชื่อมต่อและดึงข้อมูลผ่าน API มีกลไกทางเทคนิคที่ต้องบริหารจัดการดังนี้:
1. **ตัวระบุทรัพยากรระดับสากล (Uniform Resource Locator - URL)**: ที่อยู่ Endpoint เฉพาะในการเข้าถึงข้อมูล เช่นการระบุพิกัดละติจูดและลองจิจูดของสวนมังคุดหรือสวนทุเรียนเป้าหมาย
2. **รหัสผ่านการเข้าถึง (API Key)**: รหัสผ่านความปลอดภัยที่เซิร์ฟเวอร์ปลายทางกำหนดให้ส่งร่วมกับคำขอเพื่อระบุตัวตนและสิทธิ์ของผู้ใช้งาน
3. **ขีดจำกัดความเร็วคำขอ (Rate Limiting)**: มาตรการรักษาความปลอดภัยของระบบปลายทางในการจำกัดจำนวนคำขอใน 1 นาที หากส่งคำขอถี่เกินไป โปรแกรมจะถูกบล็อกและได้รับสถานะรหัสตอบกลับผิดพลาด `429 Too Many Requests`

## 3.4 กลไกการกู้คืนข้อมูลแบบทนทานต่อความล้มเหลว
การรับส่งข้อมูลผ่านโครงข่ายไร้สายภาคสนามในป่าทุเรียนหรือพื้นที่ชนบทมักเผชิญกับปัญหาสัญญาณอินเทอร์เน็ตขาดหายเป็นระยะ ๆ นักวิศวกรรมข้อมูลต้องออกแบบระบบให้มี **"ความทนทานต่อความล้มเหลว" (Fault Tolerance)** โดยมีมาตรการดังนี้:

* **ตรรกะพยายามเชื่อมต่อใหม่แบบทวีคูณ (Exponential Backoff Retry Logic)**: เมื่อตรวจพบว่าการส่งคำขอข้อมูลไปยัง API หรือ Broker ล้มเหลว โปรแกรมจะไม่ส่งคำขอถัดไปในทันที แต่จะหน่วงเวลาเป็นทวีคูณ (เช่น รอ 1 วินาที, 2 วินาที, 4 วินาที, 8 วินาที...) เพื่อป้องกันไม่ให้เครือข่ายแออัดเพิ่มขึ้นและเพิ่มโอกาสสำเร็จเมื่อระบบปลายทางกลับมาออนไลน์
* **การตรวจสอบความถูกต้องของข้อมูลภาคสนาม (Data Quality Validation)**: ตรวจสอบความถูกต้องทางตรรกะของข้อมูลเพื่อกรองสัญญาณแปลกปลอม เช่น หากค่าความชื้นสัมพัทธ์ในอากาศ (Relative Humidity - RH) มีค่าน้อยกว่า 0% หรือมากกว่า 100% ให้ทำการตัดค่านั้นทิ้งและเขียนบันทึกเตือนทันที

## 3.5 ความมั่นคงปลอดภัยของข้อมูลภาคสนาม
แม้ว่าข้อมูลการตรวจสภาพอากาศและเซ็นเซอร์วัดดินจะดูไม่มีความล่อแหลมเท่ากับข้อมูลการทำธุรกรรมทางการเงิน แต่ในระดับอุตสาหกรรมเกษตรกรรมแม่นยำ ข้อมูลเหล่านี้คือความลับทางการค้าที่มีอิทธิพลต่อผลกำไรและต้นทุน หากผู้ไม่หวังดีเจาะเข้าระบบควบคุมตรรกะ (Logic Layer) และแก้ไขค่าพารามิเตอร์พยากรณ์อากาศให้แสดงว่าฝนจะตกหนักอย่างต่อเนื่อง ระบบจะสั่งปิดการให้น้ำโดยอัตโนมัติ ส่งผลให้ผลทุเรียนขาดน้ำและร่วงหล่นสร้างความเสียหายระดับร้อยล้านบาท

### แนวปฏิบัติความปลอดภัยขั้นต่ำที่นวัตกรต้องถือปฏิบัติ:
* **การเข้ารหัสช่องทางสื่อสาร (Transport Layer Security - TLS)**: ต้องรับส่งข้อมูลผ่านโปรโตคอลปลอดภัย **HTTPS** และ **MQTTS** เสมอ เพื่อป้องกันการลักลอบดักฟังข้อมูลกลางทาง (Man-in-the-Middle Attack)
* **การแยกข้อมูลที่เป็นความลับผ่านตัวแปรสภาพแวดล้อม (Environment Variables)**: ห้ามเขียน API Key หรือรหัสผ่านฝังลงในโค้ดโปรแกรมเด็ดขาด แต่ต้องจัดเก็บไว้ในไฟล์ `.env` ที่ไม่อัปโหลดขึ้นระบบจัดเก็บโค้ด (เช่น GitHub) เพื่อความปลอดภัยเชิงระบบ

## 3.6 โค้ดคอมพิวเตอร์และแบบฝึกหัดทบทวน 3 ระดับ

เพื่อให้สอดคล้องกับยุทธศาสตร์การพัฒนาเกษตรกรอัจฉริยะ (RBRU-Praneet) โค้ดคอมพิวเตอร์และแบบฝึกหัดท้ายบทนี้ได้รับการพัฒนาและแบ่งโครงสร้างออกเป็น 3 ระดับชั้นเรียนอย่างเป็นระบบ:

### 🟩 ระดับที่ 1: มัธยมศึกษาตอนต้น (Junior High Level) - เน้นความเข้าใจพื้นฐาน
*โปรแกรมสกัดและถอดรหัสโครงสร้างข้อมูลเจสันสภาพอากาศอย่างง่าย* โค้ดนี้เน้นการฝึกทักษะการดึงข้อมูลตาม Key จาก JSON String ที่จำลองมาจากสถานีตรวจวัดอากาศภาคสนาม

#### 💻 โค้ดภาษา Python สำหรับมัธยมต้น:
```python
# โปรแกรมสกัดข้อมูลสภาพอากาศจากโครงสร้างข้อมูล JSON
# จัดทำโดย: นวัตกรน้อย RBRU-Praneet (ระดับมัธยมต้น)

import json

# 1. ข้อมูลสภาพอากาศจำลองในรูปแบบข้อความ JSON (ส่งมาจากบอร์ดเซ็นเซอร์ในสวนทุเรียน)
json_string = '{"station_id": "DURIAN-CHANTHABURI-01", "temperature": 34.2, "humidity": 78.5, "rain_probability": 85.0}'

# 2. แปลงข้อความ JSON ให้เป็นตัวแปร Dictionary ใน Python
weather_data = json.loads(json_string)

# 3. แสดงผลลัพธ์การสกัดค่าข้อมูลตามคีย์ (Key)
print("=== ระบบสกัดข้อมูลดิจิทัลเกษตรอย่างง่าย ===")
print("รหัสสถานี:", weather_data["station_id"])
print("อุณหภูมิอากาศ:", weather_data["temperature"], "องศาเซลเซียส")
print("ความชื้นสัมพัทธ์:", weather_data["humidity"], "%")
print("โอกาสที่ฝนจะตก:", weather_data["rain_probability"], "%")

# 4. ประยุกต์ใช้เงื่อนไขตรวจสอบตรรกะ
if weather_data["rain_probability"] > 80.0:
    print("วิเคราะห์ผล: [ฝนกำลังจะตกหนัก] ให้ปิดระบบรดน้ำอัตโนมัติเพื่อประหยัดพลังงาน")
else:
    print("วิเคราะห์ผล: [โอกาสฝนตกน้อย] ดำเนินการรดน้ำตามกำหนดการเดิม")
print("=========================================")
```

#### ❓ แบบฝึกหัดทบทวนระดับมัธยมต้น:
1. จงดัดแปลงตัวแปร `json_string` ให้มีโครงสร้างซับซ้อนขึ้น โดยเพิ่มคู่คีย์และค่าสำหรับเก็บ **"ชื่อพืชเศรษฐกิจ"** (`crop_name`) และวิเคราะห์ตรรกะว่าพืชดังกล่าวต้องการน้ำปริมาณมากหรือไม่
2. เขียนโค้ดสกัดค่าและแสดงผลเฉพาะค่าความชื้นสัมพัทธ์ในอากาศพร้อมตรวจสอบว่าอากาศอยู่ในสภาวะชื้นจัด (ความชื้นสูงกว่า 90%) หรือไม่

### 🟨 ระดับที่ 2: มัธยมศึกษาตอนปลาย (Senior High Level) - เน้นการดึงข้อมูลสดผ่านเครือข่ายและการดักจับข้อผิดพลาด
*โมดูลวิศวกรรมการดึงข้อมูลสภาพอากาศพยากรณ์จริงจากบริการคลาวด์* โค้ดนี้สอนทักษะการใช้งาน Library `requests` ของภาษาไพทอนเพื่อสื่อสารกับภายนอก พร้อมระบบดักจับปัญหาด้านเครือข่ายอินเทอร์เน็ตเบื้องต้น

#### 💻 โค้ดภาษา Python สำหรับมัธยมปลาย:
```python
# โปรแกรมวิศวกรรมข้อมูลดึงข้อมูลพยากรณ์อากาศผ่านเว็บเอพีไอ
# จัดทำโดย: นวัตกรสายวิทย์ RBRU-Praneet (ระดับมัธยมปลาย)

import requests

def fetch_local_weather(city_name, api_key):
    """
    ฟังก์ชันเชื่อมต่อ API เพื่อขอข้อมูลสภาพอากาศจากเซิร์ฟเวอร์ภายนอก
    """
    # 1. กำหนดที่อยู่ Endpoint และพารามิเตอร์คำขอ
    url = f"http://api.openweathermap.org/data/2.5/weather"
    params = {
        "q": city_name,
        "appid": api_key,
        "units": "metric",  # ขอข้อมูลในหน่วยเซลเซียส
        "lang": "th"       # ขอรายละเอียดสภาพอากาศเป็นภาษาไทย
    }
    
    try:
        # 2. ส่งคำขอ GET Request ผ่านเครือข่าย
        response = requests.get(url, params=params, timeout=5)
        
        # 3. ตรวจสอบรหัสสถานะการตอบกลับ (HTTP Status Code)
        # หากเกิดข้อผิดพลาด เช่น คีย์ไม่ถูกต้อง จะกระโดดไปส่วน except
        response.raise_for_status()
        
        # 4. ถอดรหัสโครงสร้างผลลัพธ์เป็น JSON Dictionary
        data = response.json()
        return data
        
    except requests.exceptions.HTTPError as http_err:
        print(f"ข้อผิดพลาด HTTP: มีปัญหาเกี่ยวกับสิทธิ์หรือคีย์เอพีไอ ({http_err})")
    except requests.exceptions.ConnectionError:
        print("ข้อผิดพลาดเครือข่าย: ไม่สามารถเชื่อมต่ออินเทอร์เน็ตได้ โปรดตรวจสอบสัญญาณ")
    except requests.exceptions.Timeout:
        print("ข้อผิดพลาดหมดเวลา: เซิร์ฟเวอร์ปลายทางตอบสนองช้าเกินไป")
    return None

# --- ส่วนทดลองรันโปรแกรมดึงข้อมูลจริงของจังหวัดจันทบุรี ---
# หมายเหตุ: ในการใช้งานจริงต้องระบุ API Key ที่ได้จากการลงทะเบียนเว็บ OpenWeather
MOCK_API_KEY = "eb7a6c986c71c4c1a5b84d9f6ab412c9" # คีย์จำลองระบบ
target_city = "Chanthaburi,TH"

print(f"เริ่มดำเนินการสืบค้นข้อมูลสภาพอากาศสำหรับพื้นที่: {target_city}")
weather_response = fetch_local_weather(target_city, MOCK_API_KEY)

if weather_response:
    print("\n--- รายงานข้อมูลสดจากสถานีบริการคลาวด์ ---")
    print(f"พื้นที่ตรวจวัด: {weather_response['name']}")
    print(f"อุณหภูมิปัจจุบัน: {weather_response['main']['temp']} °C")
    print(f"ความชื้นสัมพัทธ์: {weather_response['main']['humidity']} %")
    print(f"สภาพท้องฟ้าทั่วไป: {weather_response['weather'][0]['description']}")
    print("------------------------------------------")
else:
    print("\n[ล้มเหลว] ไม่สามารถแสดงผลข้อมูลพยากรณ์อากาศได้ ใช้ตรรกะพึ่งพาตนเองภาคสนาม")
```

#### ❓ แบบฝึกหัดทบทวนระดับมัธยมปลาย:
1. จงอธิบายสาเหตุว่าเพราะเหตุใดการส่งคำขอข้อมูลผ่านอินเทอร์เน็ตจึงต้องระบุพารามิเตอร์ **`timeout`** เสมอ และหากไม่กำหนดจะส่งผลอย่างไรต่ออุปกรณ์ควบคุมภาคสนาม
2. ดัดแปลงโค้ดข้างต้นเพื่อดึงข้อมูลสภาพอากาศของเมือง **"Rayong,TH"** และเขียนฟังก์ชันแปลงค่าอุณหภูมิที่ได้รับมาเป็นหน่วย **องศาฟาเรนไฮต์** และแสดงผลเพิ่มเติม

### 🟦 ระดับที่ 3: อุดมศึกษา (Higher Education Level) - เน้นความมั่นคงปลอดภัยและความทนทานเชิงระบบขั้นสูง
*สถาปัตยกรรมตัวจัดเก็บและเชื่อมต่อเอพีไอสภาพอากาศระดับอุตสาหกรรมพร้อมกลไก Retry และเกราะป้องกันข้อมูล* โค้ดนี้ประยุกต์ใช้การเชื่อมต่อผ่านการโหลดความลับแยกไฟล์ การทำ Retry หน่วงเวลาทวีคูณ การล้างวิเคราะห์พารามิเตอร์ชั้นสูง และการเขียนสรุปลงดิสก์อย่างทนทาน

#### 💻 โค้ดภาษา Python สำหรับอุดมศึกษา:
```python
# ระบบลำเลียงและประมวลผลข้อมูลเอพีไอสภาพอากาศระดับอุตสาหกรรม (Industrial Agri-API Pipeline)
# พัฒนาขึ้นภายใต้ยุทธศาสตร์ RBRU-Praneet (ระดับอุดมศึกษา)

import os
import time
import csv
import json
import requests
from urllib3.util import Retry
from requests.adapters import HTTPAdapter

# สมมติสถานการณ์โหลดความลับจากไฟล์สิ่งแวดล้อมจำลอง เพื่อไม่ให้รหัสผูกมัดในโค้ดหลัก
# ในสภาพงานจริงจะใช้คำสั่ง: from dotenv import load_dotenv; load_dotenv()
os.environ["AGRI_API_KEY"] = "eb7a6c986c71c4c1a5b84d9f6ab412c9"
os.environ["EXPORT_DIR"] = "./agri_data"

class RobustClimateDataEngine:
    def __init__(self, location, lat, lon):
        self.location = location
        self.lat = lat
        self.lon = lon
        self.api_key = os.getenv("AGRI_API_KEY")
        self.export_path = os.getenv("EXPORT_DIR", "./")
        
        # ปรับแต่งและสร้างโฟลเดอร์สำหรับจัดเก็บผลผลิตข้อมูล
        if not os.path.exists(self.export_path):
            os.makedirs(self.export_path)
            
        # สร้าง Session การเชื่อมต่อที่มีกลไก Retry อัจฉริยะในระดับโครงข่าย
        self.session = requests.Session()
        retries = Retry(
            total=3,                # พยายามใหม่สูงสุด 3 ครั้ง
            backoff_factor=2,       # ระยะเวลาหน่วงทวีคูณ: 2, 4, 8 วินาที
            status_forcelist=[500, 502, 503, 504], # ทำใหม่เมื่อพบคลาวด์ปลายทางพังชั่วคราว
            raise_on_status=False
        )
        self.session.mount("http://", HTTPAdapter(max_retries=retries))
        self.session.mount("https://", HTTPAdapter(max_retries=retries))

    def fetch_weather_forecast(self):
        """
        ดึงข้อมูลพยากรณ์อากาศล่วงหน้าพร้อมตรรกะการขนส่งข้อมูลที่แข็งแกร่ง
        """
        url = "https://api.openweathermap.org/data/2.5/forecast"
        params = {
            "lat": self.lat,
            "lon": self.lon,
            "appid": self.api_key,
            "units": "metric"
        }
        
        try:
            print(f"[กระบวนการ] เริ่มการดึงข้อมูลพยากรณ์สำหรับพิกัด ({self.lat}, {self.lon}) ที่สวนผลไม้ {self.location}")
            response = self.session.get(url, params=params, timeout=10)
            
            # การคัดกรองขีดจำกัดจำนวนคำขอของสิทธิ์การเชื่อมต่อ
            if response.status_code == 429:
                raise requests.exceptions.RequestException("เกิดภาวะชนขีดจำกัด Rate Limit 429 ของเซิร์ฟเวอร์")
                
            response.raise_for_status()
            return response.json()
            
        except requests.exceptions.RequestException as e:
            print(f"[ข้อผิดพลาดขั้นวิกฤต] ความล้มเหลวในการขอดึงข้อมูลภูมิอากาศ: {e}")
            return None

    def process_and_save_data(self, raw_json):
        """
        กรองวิเคราะห์ข้อมูลพยากรณ์ ล้างค่าสูญหาย และจัดบันทึกลงในโครงสร้างไฟล์ CSV
        """
        if not raw_json:
            print("[ข้อผิดพลาด] ไม่มีข้อมูลนำเข้าสำหรับการบันทึก")
            return
            
        output_file = os.path.join(self.export_path, f"{self.location}_forecast.csv")
        
        try:
            # วิเคราะห์ข้อมูลและสกัดค่าภูมิอากาศระดับลึก
            forecast_list = raw_json.get("list", [])
            
            with open(output_file, mode="w", newline="", encoding="utf-8") as file:
                writer = csv.writer(file)
                # เขียนส่วนหัวเอกสารที่ชัดเจนทางสถิติศาสตร์
                writer.writerow(["Timestamp", "DateTime", "Temperature_C", "Humidity_Pct", "Rain_Vol_3h_mm"])
                
                for entry in forecast_list:
                    timestamp = entry.get("dt")
                    dt_txt = entry.get("dt_txt")
                    temp = entry.get("main", {}).get("temp")
                    humidity = entry.get("main", {}).get("humidity")
                    
                    # ข้อมูลปริมาณฝนสะสม 3 ชั่วโมง อาจมีค่าว่างหากฝนไม่ตก
                    rain = entry.get("rain", {}).get("3h", 0.0)
                    
                    # ลอจิกการทำความสะอาดข้อมูลเบื้องต้น (Data Cleansing)
                    if temp is None or humidity is None:
                        continue # ข้ามเรกคอร์ดที่เสียหาย
                        
                    writer.writerow([timestamp, dt_txt, temp, humidity, rain])
                    
            print(f"[เสร็จสิ้น] จัดทำท่อส่งข้อมูลบันทึกลงสู่ไฟล์เรียบร้อยแล้ว: {output_file}")
            
        except IOError as io_err:
            print(f"[ข้อผิดพลาดเชิงลึก] ไม่สามารถเขียนไฟล์ลงบนหน่วยความจำได้: {io_err}")

# --- สั่งใช้งานระบบในแปลงวิจัยทุเรียนหมอนทอง จันทบุรี ---
# พิกัดอำเภอขลุง จังหวัดจันทบุรี: Lat 12.44, Lon 102.26
engine = RobustClimateDataEngine(location="Khlung_Durian_Orchard", lat=12.44, lon=102.26)
raw_forecast = engine.fetch_weather_forecast()

if raw_forecast:
    engine.process_and_save_data(raw_forecast)
else:
    print("[ระบบสำรอง] โหลดชุดข้อมูลอนุกรมเวลาท้องถิ่นเดิมมาประมวลผลแทนข้อมูลสดจากคลาวด์")
```

#### ❓ แบบฝึกหัดทบทวนระดับอุดมศึกษา:
1. จงวิเคราะห์ความแตกต่างเชิงประสิทธิภาพและเสถียรภาพของการสื่อสารแบบพุชข้อมูล (Push-based) ผ่านโพรโตคอล WebSockets เมื่อเทียบกับการร้องขอสืบค้นแบบดึงข้อมูล (Pull-based) ด้วย HTTP GET ในบริบทของระบบจัดการข้อมูลภัยแล้งในสวนผลไม้ขนาดใหญ่ที่มีการปรับเปลี่ยนข้อมูลสภาพอากาศทุกๆ 5 วินาที
2. ออกแบบโครงสร้างระบบความมั่นคง (Threat Modeling) ในการป้องกันข้อมูลการสืบค้นสถานีส่งสัญญาณของแปลงทดลองทุเรียนมูลค่าสูง เพื่อป้องกันการโจมตีประเภทตัดต่อแก้ไขชุดข้อมูล (Data Tampering) และแสดงแผนภาพการเข้ารหัสข้อมูลที่ต้องใช้ระดับอุปกรณ์ปลายทาง
