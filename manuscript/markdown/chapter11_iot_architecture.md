## 🎯 วัตถุประสงค์การเรียนรู้ (Learning Objectives)
เมื่อศึกษาบทนี้จบแล้ว ผู้เรียนจะสามารถ:
1. **วิเคราะห์ (Analyze)** ส่วนประกอบและหน้าที่ของระบบอินเทอร์เน็ตของสรรพสิ่ง (Internet of Things - IoT) ผ่านแบบจำลองสถาปัตยกรรม (Architecture) ทั้ง 3 ชั้นสำหรับการเกษตรความแม่นยำสูง
2. **ประเมิน (Evaluate)** และเลือกใช้ไมโครคอนโทรลเลอร์ (Microcontroller) ให้เหมาะสมกับเงื่อนไขทางกายภาพ แหล่งพลังงาน และชนิดเซ็นเซอร์ในพื้นที่แปลงปลูกจริง
3. **อนุพัทธ์ (Derive)** สมการคำนวณงบประมาณพลังงานไฟฟ้าและการยืดอายุการใช้งานแบตเตอรี่ในอุปกรณ์รับสัญญาณด้วยโหมดหลับลึก (Deep Sleep Mode)
4. **ออกแบบ (Design)** สถาปัตยกรรมระบบประมวลผลปลายทาง (Edge Computing) ประจำจุดตรวจวัดภาคสนาม เพื่อสร้างระบบทนทานที่ทำงานออฟไลน์ได้ 100% ยามเครือข่ายสัญญาณสื่อสารล้มเหลว
5. **สังเคราะห์ (Synthesize)** แผนผังโครงสร้างการเชื่อมต่อเครือข่ายและระบบพลังงานแสงอาทิตย์สำหรับระบบควบคุมโรงเรือนอัจฉริยะ (Smart Farming) ในพืชเศรษฐกิจมูลค่าสูง

## 11.1 นิยามและวิวัฒนาการของระบบ IoT ในภาคเกษตรกรรม

การปฏิวัติภาคเกษตรกรรมแบบดั้งเดิมสู่ยุคเกษตรกรรมอัจฉริยะพึ่งพาโครงสร้างพื้นฐานหลักคือระบบอินเทอร์เน็ตของสรรพสิ่ง ซึ่งทำหน้าที่เชื่อมต่อมิติกายภาพของไร่นาเข้ากับมิติดิจิทัลของปัญญาประดิษฐ์ ในอดีตระบบการเพาะปลูกแบบดั้งเดิมมักอาศัยการประเมินจากประสบการณ์และการสังเกตด้วยสายตาของมนุษย์ ซึ่งทำให้เกิดความคลาดเคลื่อนสูงและยากต่อการคาดการณ์ภัยแล้งหรือโรคระบาดอย่างแม่นยำ

การประยุกต์ใช้ระบบอินเทอร์เน็ตของสรรพสิ่งเข้ามาช่วยสถาปนา "เครือข่ายประสาทรับรู้ดิจิทัล" ทั่วทั้งแปลงเพาะปลูกทำให้นวัตกรสามารถแปลงคุณลักษณะทางกายภาพ เช่น ความชื้นในดิน ปริมาณไนโตรเจนที่รากดูดซึม ความเข้มแสงแดด และระดับความชื้นสัมบูรณ์ในอากาศ ให้กลายเป็นข้อมูลดิจิทัลที่ส่งผ่านอินเทอร์เน็ตได้อย่างต่อเนื่อง ยิ่งไปกว่านั้น วิวัฒนาการของเทคโนโลยีนี้ได้เปลี่ยนผ่านจากเพียงระบบควบคุมการเปิด-ปิดระยะไกลทั่วไป สู่การเป็นกลไกระบบปิดคุมตัวเองแบบอัตโนมัติที่ไม่ต้องพึ่งพามนุษย์ ช่วยจัดสรรทรัพยากรทั้งน้ำ ปุ๋ย และสารเคมีได้อย่างมีประสิทธิภาพสูงสุดในระดับตารางเมตร

## 11.2 สถาปัตยกรรม IoT 3 ชั้นในงานเกษตรกรรม (3-Layer Architecture)

การวางผังโครงสร้างของระบบอินเทอร์เน็ตของสรรพสิ่งในฟาร์มอัจฉริยะยึดตามหลักสถาปัตยกรรมอ้างอิงมาตรฐาน 3 ชั้น เพื่อจัดระเบียบการจัดเก็บ การลำเลียง และการวิเคราะห์ข้อมูลอย่างมีระบบดังนี้:

```
  ┌────────────────────────────────────────────────────────────────────────┐
  │ 3. ชั้นแอปพลิเคชัน (Application Layer)                                  │
  │ * เว็บบอร์ดแสดงผล, แดชบอร์ดกราฟิก, ระบบการแจ้งเตือน, ตัวคุมสั่งการปั๊มน้ำ│
  └────────────────────────────────────────────────────────────────────────┘
                                     ▲
                                     │ (โพรโตคอล MQTT / HTTP / REST API)
                                     ▼
  ┌────────────────────────────────────────────────────────────────────────┐
  │ 2. ชั้นโครงข่ายสื่อสาร (Network Layer)                                  │
  │ * สัญญาณอินเทอร์เน็ตไร้สาย, เสาสัญญาณระยะไกล LoRaWAN, เกตเวย์รับ-ส่ง     │
  └────────────────────────────────────────────────────────────────────────┘
                                     ▲
                                     │ (การส่งผ่านสายสัญญาณ / สัญญานอนาล็อก)
                                     ▼
  ┌────────────────────────────────────────────────────────────────────────┐
  │ 1. ชั้นประสาทรับรู้ (Perception Layer)                                   │
  │ * เซ็นเซอร์วัดดิน, หัววัดระดับน้ำ, เครื่องวัดแสง, วาล์วไฟฟ้าอัจฉริยะ       │
  └────────────────────────────────────────────────────────────────────────┘
```

### 11.2.1 ชั้นประสาทรับรู้ (Perception Layer)
เปรียบเสมือนดวงตาและผิวสัมผัสของแปลงเพาะปลูก ทำหน้าที่เก็บรวบรวมค่าพารามิเตอร์จริงทางชีวภาพและกายภาพ ประกอบไปด้วยเซ็นเซอร์วัดค่าสภาพแวดล้อม เช่น หัววัดอุณหภูมิและความชื้นสัมบูรณ์ ตัววัดความชื้นดินชนิดคลื่นความถี่สูง ตัววัดค่าธาตุอาหารไนโตรเจน-ฟอสฟอรัส-โพแทสเซียม รวมถึงแอคทูเอเตอร์หรือตัวแปลงพลังงานไฟฟ้าให้เป็นความเคลื่อนไหวทางกล เช่น โซลินอยด์วาล์วน้ำ รีเลย์สั่งเปิดพัดลมระบายอากาศ และปั๊มสารอาหารพืช

### 11.2.2 ชั้นโครงข่ายสื่อสาร (Network Layer)
เปรียบเสมือนระบบประสาทกลาง ทำหน้าที่ส่งผ่านชุดข้อมูลตัวเลขที่เก็บรวบรวมได้จากชั้นประสาทรับรู้ไปยังระบบประมวลผลส่วนกลาง ชั้นนี้ประกอบด้วยฮาร์ดแวร์สถานีสื่อสาร อุปกรณ์เกตเวย์ (Gateway) ประจำแปลง และโพรโตคอลการรับส่งสัญญาณแบบไร้สาย (Wireless) ระยะสั้นและระยะไกล เช่น สัญญาณไวไฟ (Wi-Fi), บลูทูธ (Bluetooth), ลอราแวน (LoRaWAN) สำหรับพื้นที่สวนป่าห่างไกล และเอ็นบีไอโอที (NB-IoT) ผ่านเครือข่ายมือถือของประเทศ

### 11.2.3 ชั้นแอปพลิเคชัน (Application Layer)
เปรียบเสมือนสมองใหญ่ทำหน้าที่วิเคราะห์ ประมวลผล และนำเสนอข้อมูลสู่ผู้ใช้งาน ชั้นนี้ประกอบด้วยคลาวด์เซิร์ฟเวอร์ ระบบฐานข้อมูลอนุกรมเวลาเพื่อเก็บประวัติค่าเซ็นเซอร์ และหน้ากากแอปพลิเคชันหรือแดชบอร์ด (Dashboard) ที่แสดงผลกราฟระดับความชื้นในดิน ทิศทางลม ปริมาณน้ำฝน ตลอดจนปุ่มสำหรับให้เกษตรกรสามารถควบคุมเปิดปิดวาล์วน้ำระยะไกลผ่านสมาร์ทโฟนได้โดยสะดวก

## 11.3 การเลือกไมโครคอนโทรลเลอร์และการบริหารพลังงาน (Hardware Selection & Power)

หัวใจสำคัญของการประกอบสถานีตรวจวัดสัญญาณภาคสนามคือตัวประมวลผลขนาดจิ๋วหรือไมโครคอนโทรลเลอร์ ซึ่งต้องมีสมรรถนะเพียงพอในการอ่านค่าดิจิทัลและส่งข้อมูลแบบไร้สาย แต่ต้องประหยัดพลังงานมากพอที่จะรันด้วยโซลาร์เซลล์ (Solar Cell) หรือแบตเตอรี่ในสวนทุเรียนที่ไม่มีไฟฟ้าเข้าถึงเป็นเวลาหลายปี

```
  ┌────────────────────────────────────────────────────────────────────────────────────────┐
  │                          ตารางเปรียบเทียบคุณสมบัติไมโครคอนโทรลเลอร์                       │
  ├──────────────────────┬────────────────────────┬────────────────────────┬───────────────┤
  │      คุณลักษณะ       │       Arduino Uno      │          ESP32         │  STM32 (Blue) │
  ├──────────────────────┼────────────────────────┼────────────────────────┼───────────────┤
  │ ความเร็วสัญญานาฬิกา    │ 16 MHz                 │ 160 - 240 MHz (Dual)   │ 72 MHz        │
  │ หน่วยความจำแรม (RAM)  │ 2 KB                   │ 520 KB                 │ 20 KB         │
  │ การสื่อสารในตัว        │ ไม่มี (ต้องซื้อโมดูลเพิ่ม) │ Wi-Fi & Bluetooth ในตัว │ ไม่มี         │
  │ การใช้พลังงานหลับลึก    │ ~ 15 mA (สูง)          │ ~ 15 uA (ต่ำมากวิกฤต)   │ ~ 2 uA (ต่ำสุด)│
  │ รองรับภาษาโปรแกรม    │ C++ เท่านั้น            │ C++, MicroPython, Rust │ C, C++        │
  └──────────────────────┴────────────────────────┴────────────────────────┴───────────────┘
```

ไมโครคอนโทรลเลอร์บอร์ด **ESP32** ได้กลายเป็นมาตรฐานการใช้งานทางการเกษตรเนื่องจากรองรับโปรโตคอลการสื่อสารไร้สายในชิปชิ้นเดียว และสามารถทำงานร่วมกับระบบปฏิบัติการคอมพิวเตอร์ย่อยรวมถึงไมโครไพทอน (MicroPython) ซึ่งช่วยย่นเวลาการเขียนโค้ดและบำรุงรักษาระบบของผู้ดูแลฟาร์ม

### 11.3.1 คณิตศาสตร์การประเมินงบประมาณพลังงาน (Energy Budget Mathematics)
เพื่อรับประกันว่าตัวสถานีตรวจวัดสัญญาณจะไม่ดับลงกลางคันในคืนที่มีฝนตกติดต่อกันเป็นสัปดาห์ในจันทบุรี นวัตกรต้องคำนวณสเกลงบประมาณความต้องการพลังงานอย่างรัดกุมผ่านสมการการแผ่พลังงานเฉลี่ย

สมมติให้อุปกรณ์ทำงานเป็นรอบเวลา (Duty Cycle) ทุก ๆ 1 ชั่วโมง (3600 วินาที):
* **สถานะทำงาน (Active State):** อุปกรณ์ตื่นขึ้นมาอ่านค่าเซ็นเซอร์และส่งค่าผ่านสัญญาณไวไฟ ใช้กระแสไฟฟ้า $I_{\text{active}} = 120 \text{ mA}$ เป็นระยะเวลา $t_{\text{active}} = 5 \text{ seconds}$
* **สถานะหลับลึก (Deep Sleep State):** อุปกรณ์ดับสัญญานาฬิกาและโมดูลส่งคลื่นวิทยุเหลือเพียงวงจรจับเวลารักษาเวลาหลัก ใช้กระแสไฟฟ้า $I_{\text{sleep}} = 15 \ \mu\text{A}$ ($0.015 \text{ mA}$) เป็นระยะเวลา $t_{\text{sleep}} = 3595 \text{ seconds}$
* **แรงดันไฟฟ้าระบบปฏิบัติการ:** $V = 3.3 \text{ V}$

#### 1. คำนวณประจุไฟฟ้าทั้งหมดที่ต้องการในหนึ่งรอบชั่วโมง ($Q_{\text{cycle}}$):
สูตรคำนวณปริมาณประจุไฟฟ้า (แอมแปร์-วินาที หรือ A·s):

$$Q_{\text{cycle}} = (I_{\text{active}} \times t_{\text{active}}) + (I_{\text{sleep}} \times t_{\text{sleep}})$$

แปลงหน่วยกระแสให้อยู่ในรูปแอมแปร์ (A):
$$I_{\text{active}} = 0.120 \text{ A}$$
$$I_{\text{sleep}} = 0.000015 \text{ A}$$

แทนค่าในสมการ:
$$Q_{\text{cycle}} = (0.120 \times 5) + (0.000015 \times 3595) = 0.600 + 0.0539 = 0.6539 \text{ A}\cdot\text{s}$$

#### 2. คำนวณความต้องการกระแสเฉลี่ยต่อชั่วโมง ($I_{\text{average}}$):
$$I_{\text{average}} = \frac{Q_{\text{cycle}}}{T_{\text{total\_seconds}}} = \frac{0.6539 \text{ A}\cdot\text{s}}{3600 \text{ s}} \approx 0.0001816 \text{ A} \approx 0.1816 \text{ mA}$$

#### 3. คำนวณอายุการใช้งานจากแบตเตอรี่ชนิดลิเทียมไอออนฟอสเฟตขนาดประจุใช้งานได้ $3000 \text{ mAh}$ ($3 \text{ Ah}$):
อายุการใช้งานเป็นจำนวนชั่วโมง ($T_{\text{life}}$):

$$T_{\text{life}} = \frac{\text{Battery Capacity (Ah)}}{I_{\text{average}} (A)} = \frac{3.0}{0.0001816} \approx 16,520 \text{ hours}$$

แปลงหน่วยเป็นจำนวนวันและปี:
$$\text{Days} = \frac{16,520}{24} \approx 688 \text{ days} \approx 1.88 \text{ years}$$

> [!TIP]
> เปรียบเทียบผลลัพธ์ยามไม่ใช้ระบบโหมดหลับลึก: หากทิ้งให้ ESP32 ทำงานแบบตื่นตลอดเวลา ($120 \text{ mA}$) แบตเตอรี่จะอยู่ได้เพียง $3000 \text{ mAh} / 120 \text{ mA} = 25 \text{ hours}$ (เพียงประมาณ 1 วันเท่านั้น!) สมการคณิตศาสตร์นี้พิสูจน์ให้เห็นอย่างชัดเจนว่าโหมดหลับลึกคือตัวตัดสินขีดความสามารถการติดตั้งใช้งานเซ็นเซอร์ระยะไกลได้อย่างแท้จริง

## 11.4 สถาปัตยกรรม Edge Computing ในระบบ IoT เกษตรเพื่อความทนทาน

ในแปลงเกษตรกรรม ปัญหาการสูญเสียการเชื่อมต่อเครือข่ายเป็นสิ่งที่เกิดขึ้นบ่อยครั้งจากพายุฝน ลมกระโชกแรง หรือแม้แต่สัญญาณขัดข้องจากผู้ให้บริการเครือข่าย หากระบบ IoT ถูกออกแบบให้ส่งผ่านค่าเซ็นเซอร์ดิบขึ้นไปประมวลผลบนคลาวด์เพื่อตัดสินใจรดน้ำทุกครั้ง เมื่อใดก็ตามที่อินเทอร์เน็ตดับลง แปลงปลูกพืชอาจขาดน้ำจนเหี่ยวแห้งตาย หรือในทางกลับกันปั๊มน้ำอาจทำงานค้างจนทำให้น้ำท่วมโคนต้นทุเรียนจนเกิดโรคเน่าได้

การแก้ไขปัญหานี้ต้องอาศัยสถาปัตยกรรมระบบประมวลผลปลายทาง ซึ่งเป็นการย้ายตรรกะการตัดสินใจทางวิศวกรรมลงมาไว้ที่ตัวไมโครคอนโทรลเลอร์ภายในพื้นที่ (Local Edge Control) ดังนี้:

```
  [ สถานการณ์เครือข่ายปกติ ]
  เซ็นเซอร์ดิน ────────> บอร์ด ESP32 (วิเคราะห์ข้อมูลที่ขอบ) ────────> สั่งรีเลย์รดน้ำท้องถิ่น (ทันที)
                                 │
                                 └─ (ผ่านสัญญาณปกติ) ─> ส่งสถิติขึ้นคลาวด์ Dashboard

  [ สถานการณ์อินเทอร์เน็ตล่ม ]
  เซ็นเซอร์ดิน ────────> บอร์ด ESP32 (ทำงานออฟไลน์ 100%) ─────────> สั่งรีเลย์รดน้ำท้องถิ่น (เสถียร)
                                 │
                                 └─ (อินเทอร์เน็ตหลุด) ─> บันทึกลงหน่วยความจำสำรองชั่วคราว (SD Card)
```

1. **โหมดการควบคุมแบบกระจายศูนย์ (Distributed Control Logic):** อุปกรณ์ขอบปลายทางจะอ่านและวิเคราะห์ค่าดินเทียบกับค่าขีดจำกัดวิกฤตที่ฝังอยู่ในชิปโดยตรง หากความชื้นต่ำกว่าค่าปลอดภัย ระบบจะเปิดปั๊มน้ำในทันทีโดยไม่ต้องส่งคำขอไปคลาวด์เซิร์ฟเวอร์
2. **ระบบบันทึกความล้มเหลวแบบยืดหยุ่น (Offline Storage Resilience):** ยามสัญญาณการเชื่อมต่อขาดหาย บอร์ดจะไม่ละทิ้งค่าเซ็นเซอร์ แต่จะสลับไปโหมดกักเก็บข้อมูลท้องถิ่นโดยเขียนไฟล์ลงในโมดูลหน่วยความจำภายนอกชั่วคราว และเมื่อโปรแกรมตรวจพบว่าสัญญาณเครือข่ายกลับมาปกติ จึงจะสตรีมส่งไฟล์ย้อนหลังกลับขึ้นระบบคลาวด์โดยสมบูรณ์ป้องกันปัญหาข้อมูลขาดหายในประวัติการเพาะปลูก

## 11.5 การสังเคราะห์กรณีศึกษา: การวางผังระบบควบคุมโรงเรือนอัจฉริยะ

*การศึกษาเชิงลึก:* การออกแบบระบบควบคุมโรงเรือนเพาะปลูกพืชผลเมล่อนอัจฉริยะขนาดพื้นที่ $10 \times 20$ เมตร ณ แปลงวิจัยประยุกต์ ศูนย์ RBRU-Praneet

### 1. การกำหนดตำแหน่งและการจัดวางตัวตรวจวัด (Sensor Node Placement):
การประเมินเชิงพื้นที่ของความผันผวนทางชีวภาพชี้วัดว่า อุณหภูมิและความชื้นภายในโรงเรือนแบบปิดไม่เคยคงที่เท่ากันทุกจุด นวัตกรจึงต้องจัดวางจุดตรวจวัดเป็นแบบแบ่งโซน:
* **ระดับความสูง:** วางเซ็นเซอร์ DHT22 ณ ความสูงสองระดับ คือระดับยอดใบพืช (เพื่อดูอัตราการคายน้ำและการค้างของน้ำค้าง) และระดับเหนือผิวดินเล็กน้อย
* **มิติเชิงราบ:** ติดตั้งโหนดเก็บค่าความชื้นดินแบบหัวตรวจความต้านทานสัมพัทธ์รวม 3 จุด (หน้าประตูโรงเรือน, ส่วนแกนกลาง, และท้ายโรงเรือนติดพัดลมระบายอากาศ) เพื่อสืบค้นจุดอับน้ำ

### 2. เกตเวย์และระบบสำรองพลังงานโฟโตโวลตาอิก (Gateway & Solar Integration):
* **ตัวส่งต่อ (Gateway):** บอร์ดตัวส่งใช้เรสเบอร์รีพายที่บรรจุระบบซอฟต์แวร์กลาง คอยรับสัญญาณวิทยุจากบอร์ดเซ็นเซอร์ย่อยด้วยคลื่นความถี่เฉพาะตัว และทำหน้าที่เป็นตัวสะสมข้อมูลส่งต่อผ่านระบบสายแลน (LAN) สู่เครือข่ายหลักของมหาวิทยาลัย
* **การออกแบบระบบแผงโซลาร์เซลล์ประจำโหนดสนาม:**
  จากการวิเคราะห์การกินกระแสไฟเฉลี่ย $0.1816 \text{ mA}$ ที่แรงดันไฟระบบ $3.3 \text{ V}$ พลังงานพลังงานเฉลี่ยที่ใช้ต่อวันคือ:
  
  $$E_{\text{day}} = 0.1816 \text{ mA} \times 3.3 \text{ V} \times 24 \text{ Hours} \approx 14.38 \text{ mWh} \approx 0.01438 \text{ Wh}$$
  
  เพื่อชดเชยการสูญเสียในระบบชาร์จเจอร์และเตรียมพร้อมกรณีฟ้ามืดติดต่อกัน 5 วันโดยไม่มีแสงแดด ระบบสำรองข้อมูลกำหนดให้ติดตั้งชุดสะสมประจุขนาดกำลังการจ่ายกำลังอย่างน้อย $1 \text{ W}$ โฟโตโวลตาอิก (Photovoltaic) ร่วมกับชุดแบตเตอรี่ชนิดลิเทียมความจุสูง $2000 \text{ mAh}$ ซึ่งมีปริมาณกำลังไฟฟ้าสะสมล้นเหลือพอต่อการจ่ายระบบได้ยาวนานตลอดปีอย่างเสถียร

## 11.6 โค้ดคอมพิวเตอร์และแบบฝึกหัดทบทวน 3 ระดับ

ในการติดตั้งสร้างระบบซอฟต์แวร์ประมวลผลเครือข่าย การจำลองและการเขียนโปรแกรมสามารถถูกแบ่งระดับออกเป็นโครงสร้างย่อย 3 ขั้นการเรียนรู้:

### 🟩 ระดับที่ 1: มัธยมศึกษาตอนต้น (Junior High Level) - สคริปต์ตรวจจับค่าความชื้นดินและสั่งการปั๊มน้ำจำลอง
*คำอธิบาย:* โค้ดนี้ใช้ภาษาไพทอนจำลองการอ่านค่าพารามิเตอร์จำลองความชื้นดินในแปลงผักส่วนโรงเรียนผ่านอินพุตพอร์ตแบบอนาล็อก และประเมินค่าเพื่อแสดงข้อความเตือนภัยและเปิดรีเลย์ปั๊มน้ำผ่านจอภาพคอนโซลเบื้องต้น

#### 💻 โค้ดภาษา Python สำหรับมัธยมต้น:
```python
# โปรแกรมจำลองการอ่านข้อมูลเซ็นเซอร์แปลงดินและคุมวาล์วน้ำท้องถิ่น
# จัดทำโดย: นวัตกรน้อย RBRU-Praneet (ระดับมัธยมต้น)

import time
import random

# กำหนดเกณฑ์ขั้นต่ำสำหรับเตือนดินแห้ง (หากความชื้นต่ำกว่าร้อยละ 40)
DRY_THRESHOLD_PCT = 40

print("--- เริ่มต้นการทำงานระบบควบคุมปั๊มน้ำอัจฉริยะ (จำลอง) ---")

# ลูปจำลองการอ่านค่าเซ็นเซอร์ต่อเนื่อง 5 รอบเวลา
for epoch in range(1, 6):
    print(f"\nรอบสแกนที่ {epoch} ของวัน:")
    
    # จำลองการวัดค่าความชื้นจากเซ็นเซอร์ดินแบบสุ่มเลขทศนิยมเปอร์เซ็นต์
    simulated_moisture = random.uniform(20.0, 75.0)
    print(f"ค่าความชื้นในดินที่อ่านได้: {simulated_moisture:.2f}%")
    
    # ตรรกะระบบควบคุมการตัดสินใจ
    if simulated_moisture < DRY_THRESHOLD_PCT:
        print(">> สถานะดิน: [แห้งจัดวิกฤต!]")
        print(">> การกระทำ: สั่งปล่อยกระแสไฟฟ้ากระตุ้นวาล์วรีเลย์ -> [เปิดปั๊มน้ำรดน้ำ]")
    else:
        print(">> สถานะดิน: [ความชื้นเพียงพอปกติ]")
        print(">> การกระทำ: คงสัญญาสั่งการเดิม -> [ปิดปั๊มน้ำไว้]")
        
    time.sleep(1.5)  # เว้นจังหวะรอบคำนวณถัดไป

print("\n--- เสร็จสิ้นการทำงานลูปจำลองระบบ ม.ต้น ---")
```

#### ❓ แบบฝึกหัดทบทวนระดับมัธยมต้น:
1. หากนวัตกรต้องการปรับเปลี่ยนเงื่อนไขให้เหมาะกับหน้าดินทรายปนเหนียวที่อุ้มน้ำได้ต่ำและต้องการรักษาความชื้นเฉลี่ยไว้ที่ระดับไม่น้อยกว่า $55\%$ นักเรียนต้องแก้ไขโค้ดโปรแกรมในบรรทัดใด และต้องเขียนปรับปรุงตัวเลขอย่างไร
2. การใส่ค่าดีเลย์หน่วงเวลา `time.sleep(1.5)` มีประโยชน์อย่างไรในมุมมองวิศวกรรมการสลายอุณหภูมิและการประหยัดโหลดประมวลผลไมโครชิป

---

### 🟨 ระดับที่ 2: มัธยมศึกษาตอนปลาย (Senior High Level) - โค้ด MicroPython สำหรับ ESP32 เพื่อควบคุมโหมดหลับลึกและส่งข้อมูลแบบ JSON
*คำอธิบาย:* โค้ดนี้ถูกสังเคราะห์ขึ้นเพื่อใช้รันจริงบนสถาปัตยกรรมบอร์ด ESP32 ด้วยภาษาไมโครไพทอนเพื่อจัดการบริหารสเกลพลังงานผ่านการหลับลึก และการห่อหุ้มตัวแปรเซ็นเซอร์ด้วยรูปแบบพิกัดแบบเจซอน (JSON) เพื่อเตรียมการนำไปส่งผ่านเครือข่าย

#### 💻 โค้ดภาษา MicroPython สำหรับมัธยมปลาย:
```python
# โปรแกรมควบคุมบริหารพลังงานอัจฉริยะและรวมดาต้าเพย์โหลดสำหรับบอร์ด ESP32
# จัดทำโดย: นวัตกรระดับมัธยมปลาย RBRU-Praneet

import machine
import time
import ujson

# 1. กำหนดโครงสร้างขาเซ็นเซอร์และการควบคุม
# ขา ADC สำหรับเซ็นเซอร์วัดดิน (ใช้ GPIO 34 บน ESP32)
adc_soil = machine.ADC(machine.Pin(34))
adc_soil.atten(machine.ADC.ATTN_11DB)  # ตั้งขนาดการรับแรงดันไฟสูงสุด 3.3V

# รีเลย์ควบคุมสำหรับต่อไฟปั๊มน้ำ (ใช้ GPIO 26)
relay_pump = machine.Pin(26, machine.Pin.OUT)

print("--- บอร์ดตื่นจากสภาวะสลีป ดำเนินงานเก็บตัวเลขสัญญาณ ---")

# 2. อ่านค่าทางไฟฟ้าแปลงค่าสัญญานดิจิทัล (ความละเอียด 12 บิต: ช่วง 0 - 4095)
raw_val = adc_soil.read()
# คำนวณร้อยละเชิงเส้นกลับข้าง (สมมติให้ค่า 4095 คือดินแห้งสนิท และ 0 คือในถังน้ำเปียกสุด)
moisture_percent = ((4095 - raw_val) / 4095) * 100

# 3. จัดทำตรรกะสั่งการออฟไลน์ ณ ปลายทางขอบ (Local Decision)
if moisture_percent < 45.0:
    relay_pump.on()  # สั่งรีเลย์เปิดปั๊มน้ำ
    pump_status = "ACTIVE"
else:
    relay_pump.off() # ปิดปั๊ม
    pump_status = "INACTIVE"

# 4. ประกอบชุดข้อมูลสรุปในรูปแบบ JSON Payload เพื่อพร้อมสำหรับการลำเลียง
telemetry_data = {
    "node_id": "RBRU-NODE-01",
    "raw_analog": raw_val,
    "soil_moisture_pct": round(moisture_percent, 2),
    "relay_pump_state": pump_status,
    "v_battery": 3.28  # ค่าแบตเตอรี่จำลองส่งผ่านตรวจสอบ
}

json_string = ujson.dumps(telemetry_data)
print("สรุปรูปแบบข้อมูลจัดทำเสร็จสิ้น:")
print(json_string)

# หน่วงเวลาสั้น ๆ ให้น้ำไหลเข้าระบบและบันทึกผลเสร็จสมบูรณ์
time.sleep(2)

# 5. สั่งการบอร์ดเข้าสู่โหมดหลับลึกเพื่อประหยัดพลังงานเป็นเวลา 15 นาที
print("บอร์ดกำลังเข้าสู่โหมด Deep Sleep เป็นเวลา 15 นาที...")
machine.deepsleep(15 * 60 * 1000)  # แปลงนาทีเป็นมิลลิวินาที
```

#### ❓ แบบฝึกหัดทบทวนระดับมัธยมปลาย:
1. การระบุเรียกใช้งานพารามิเตอร์คำสั่ง `adc_soil.atten(machine.ADC.ATTN_11DB)` มีความจำเป็นอย่างไรในการประมวลผลสัญญาณกายภาพทางไฟฟ้าของบอร์ด ESP32 เพื่อป้องการวงจรภายในเสียหาย
2. หากบอร์ดตื่นจากการทำ `machine.deepsleep` ระบบจะเริ่มรันจากบรรทัดแรกเริ่มใหม่เสมอหรือจะทำต่อจากคำสั่งบรรทัดสุดท้ายของการหลับ จงอธิบายลอจิกการอัปเดตตัวแปรภายในหน่วยความจำแรม

---

### 🟦 ระดับที่ 3: อุดมศึกษา (Higher Education Level) - โครงสร้างระบบประมวลผลปลายทางและระบบสำรองข้อมูลยามเครือข่ายล่ม
*คำอธิบาย:* โค้ดประยุกต์เชิงวิทยาศาสตร์ระบบอัจฉริยะนี้ พัฒนาด้วยเทคนิคการประมวลผลเชิงวัตถุ (Object-Oriented Programming - OOP) เพื่อสร้างเป็นคลาสโครงข่ายสถานีหลัก สามารถตรวจเช็คสัญญาณระบบไร้สายในสวนทุเรียน หากตรวจพบการเชื่อมต่อเครือข่ายล้มเหลว จะสลับสวิตช์มาจัดเก็บไฟล์ย้อนหลังในรูปแบบ JSON ลงดิสก์ฐานข้อมูลจำลอง และทำงานส่งชดเชยอัตโนมัติทันทีที่สัญญาณระบบปกติ พร้อมทั้งรายงานสถิติผ่านการล็อกเกอร์ระบบอย่างครบวงจร

#### 💻 โค้ดภาษา Python สำหรับระดับอุดมศึกษา:
```python
"""
ระบบจำลองวิศวกรรมฝั่งสถานีปลายทางขอบอัจฉริยะแบบทำงานออฟไลน์ทนทาน (Offline Resilient Agri-Node)
จัดทำสำหรับแบบศึกษาทดลองวิจัยระดับอุดมศึกษา ศูนย์วิทยาศาสตร์ข้อมูล RBRU-Praneet 2026
"""

import os
import json
import random
import time
import logging

# ตั้งค่าสถานะแจ้งเตือนข้อผิดพลาดระบบ
logging.basicConfig(level=logging.INFO, format="%(asctime)s [%(levelname)s] %(message)s")
logger = logging.getLogger("AgriNodeSystem")

class LocalStorageResilience:
    def __init__(self, storage_path="local_buffer.json"):
        self.storage_path = storage_path
        
    def save_offline(self, data):
        """
        บันทึกข้อมูลเทเลเมทรีของฟาร์มเก็บเข้าไฟล์ชั่วคราวยามไม่มีเครือข่ายสัญญาณ
        """
        existing_data = []
        if os.path.exists(self.storage_path):
            try:
                with open(self.storage_path, "r", encoding="utf-8") as f:
                    existing_data = json.load(f)
            except Exception as e:
                logger.error(f"เกิดข้อผิดพลาดในการเปิดไฟล์บันทึกออฟไลน์: {e}")
                existing_data = []
                
        existing_data.append(data)
        
        try:
            with open(self.storage_path, "w", encoding="utf-8") as f:
                json.dump(existing_data, f, indent=4, ensure_ascii=False)
            logger.warning(f"บันทึกข้อมูลออฟไลน์สำรองเรียบร้อย (จำนวนสะสม: {len(existing_data)} เรคคอร์ด)")
        except Exception as e:
            logger.critical(f"ล้มเหลวในการเขียนข้อมูลลงหน่วยความจำถาวร: {e}")

    def read_and_clear(self):
        """
        อ่านข้อมูลออฟไลน์ที่บันทึกค้างไว้ทั้งหมดออกและทำการล้างไฟล์เพื่อพร้อมส่งกลับคลาวด์
        """
        if not os.path.exists(self.storage_path):
            return []
        
        try:
            with open(self.storage_path, "r", encoding="utf-8") as f:
                data = json.load(f)
            os.remove(self.storage_path)
            logger.info(f"ดึงข้อมูลออฟไลน์เรียบร้อย ล้างประจุไฟล์หน่วยความจำสำรองชั่วคราวสำเร็จ")
            return data
        except Exception as e:
            logger.error(f"ไม่สามารถดำเนินการล้างไฟล์หน่วยความจำสำรองได้: {e}")
            return []

class ResilientAgriNode:
    def __init__(self, node_id="RBRU-PRANEET-NODE-A4"):
        self.node_id = node_id
        self.storage = LocalStorageResilience()
        self.is_network_online = True  # สถานะเครือข่ายจำลอง

    def check_network_status(self):
        """
        วิเคราะห์ตรวจสอบสัญญาณเชื่อมต่อจริงเชิงโครงสร้างเครือข่าย (จำลองสุ่มสถานะเพื่อการทดสอบ)
        """
        # สุ่มให้อินเทอร์เน็ตล่มเป็นบางช่วง เพื่อเลียนแบบสภาพอากาศพายุฝนสวนทุเรียนจันทบุรีจริง
        self.is_network_online = random.choice([True, True, True, False])
        if self.is_network_online:
            logger.info("สถานะการเชื่อมต่อเกตเวย์: [เชื่อมต่อปกติ - ออนไลน์]")
        else:
            logger.warning("สถานะการเชื่อมต่อเกตเวย์: [สัญญาณล้มเหลว - ออฟไลน์!]")
        return self.is_network_online

    def harvest_sensor_data(self):
        """
        เก็บเกี่ยวตัวเลขพารามิเตอร์จากเซ็นเซอร์ดิน DHT22 และแบตเตอรี่ในสวนผสม
        """
        return {
            "timestamp": int(time.time()),
            "node_id": self.node_id,
            "soil_moisture": round(random.uniform(30.0, 80.0), 2),
            "air_temp_c": round(random.uniform(25.0, 38.0), 2),
            "air_humidity_pct": round(random.uniform(50.0, 95.0), 2),
            "battery_volt": round(random.uniform(3.1, 4.2), 2)
        }

    def push_to_cloud(self, payload):
        """
        จำลองการประมวลผลยิงข้อมูลผ่านโปรโตคอล API ไปเก็บยังฐานข้อมูลบนคลาวด์เซิร์ฟเวอร์
        """
        logger.info(f"ยิงข้อมูลสำเร็จเข้าสู่ Cloud Server -> ข้อมูลหลัก: ความชื้นดิน {payload['soil_moisture']}%")
        return True

    def process_cycle(self):
        """
        ประมวลผลกระบวนการจัดการข้อมูลอัจฉริยะหนึ่งรอบเวลา
        """
        # 1. การเก็บข้อมูล
        current_data = self.harvest_sensor_data()
        
        # 2. ตรวจสอบเงื่อนไขเครือข่ายลำเลียง
        if self.check_network_status():
            # ก. หากระบบออนไลน์: ส่งต่อข้อมูลปัจจุบันขึ้นสู่ระบบคลาวด์
            self.push_to_cloud(current_data)
            
            # ข. ตรวจเช็คว่ามีข้อมูลออฟไลน์ค้างเก่าในแฟ้มจัดเก็บหรือไม่ เพื่อส่งล้างย้อนหลัง
            backlog = self.storage.read_and_clear()
            if backlog:
                logger.info(f"ตรวจพบข้อมูลออฟไลน์ตกค้าง {len(backlog)} ชุด ดำเนินการยิงชดเชยย้อนหลัง...")
                for old_data in backlog:
                    self.push_to_cloud(old_data)
                    time.sleep(0.2)  # ป้องกันปัญหาระบบเซิร์ฟเวอร์ตอบรับไม่ทัน
        else:
            # ค. หากเครือข่ายเสียหาย: หลีกเลี่ยงสัญญาณข้อความล้มเหลว ย้ายการทำงานลงที่ระบบจัดเก็บขอบชั่วคราว
            logger.error("อินเทอร์เน็ตใช้งานไม่ได้! สลับการประมวลผลเข้าโหมด Local Storage Resilience ทันที")
            self.storage.save_offline(current_data)

if __name__ == "__main__":
    node = ResilientAgriNode()
    
    print("\n================ เริ่มต้นทดสอบจำลองเหตุการณ์จริง 6 คาบเวลา ================")
    for cycle in range(1, 7):
        logger.info(f"--- เริ่มต้นคาบการทำงานของระบบที่ {cycle} ---")
        node.process_cycle()
        time.sleep(1.0)
    print("==========================================================================")
    
    # ดำเนินการลบทำความสะอาดระบบไฟล์เก็บข้อมูลทดสอบภายนอก
    if os.path.exists("local_buffer.json"):
        os.remove("local_buffer.json")
```

## 11.7 บทสรุปและคำถามทบทวนเชิงวิเคราะห์ (Chapter Summary & Analytical Exercises)

### 📌 บทสรุปของบทเรียน:
การพัฒนาและประยุกต์ใช้งานระบบเกษตรกรรมอัจฉริยะความแม่นยำสูงจำเป็นต้องพึ่งพาสถาปัตยกรรมระบบอินเทอร์เน็ตของสรรพสิ่งที่เป็นระเบียบ โดยแบ่งระดับเป็น 3 เลเยอร์หลักตั้งแต่การเก็บค่าพารามิเตอร์ การส่งคลื่นวิทยุไร้สาย ไปจนถึงการจัดแสดงบนแดชบอร์ด ในการลงปฏิบัติการทำงานจริงการเลือกสรรฮาร์ดแวร์โดยเน้นบอร์ดประมวลผลอเนกประสงค์ ESP32 และการใช้เทคนิคคำนวณสมดุลระดับกระแสในโหมดหลับลึก ถือเป็นเงื่อนไขที่กำหนดเส้นแบ่งความสำเร็จของระยะความคงทนของระบบ

นอกจากนี้ การวางแผนสถาปัตยกรรมให้มีระบบประมวลผลปลายทางร่วมอยู่ในแปลงเพาะปลูกและจัดตั้งกลไกพื้นที่กักเก็บข้อมูลสำรองยามสัญญาณสื่อสารขัดข้อง คือแนวนโยบายทางวิศวกรรมที่ป้องกันระบบฟาร์มเมล่อนและพืชผลล่มสลาย ทำให้เทคโนโลยีอัจฉริยะสามารถทำงานได้อย่างคงเส้นคงวาและส่งต่อผลผลิตที่มีคุณภาพสูงได้อย่างยั่งยืนตลอดไป

### ❓ คำถามทบทวนเชิงวิเคราะห์เชิงรวม (Comprehensive Analytical Questions):

1. **(ระดับวิเคราะห์ - Analyze)** 
   จงวิเคราะห์ความแตกต่างเชิงเทคนิคในด้านการใช้โครงสร้างเครือข่ายระยะทางการรับส่งข้อมูล ความทนทานต่อต้นไม้หนาทึบในสวนผลไม้ และสัดส่วนอัตรากินกระแสไฟฟ้าระหว่างสถาปัตยกรรมที่ต่อเชื่อมผ่าน Wi-Fi วิ่งตรงสู่เกตเวย์ และสถาปัตยกรรมสื่อสารแบบ LoRaWAN ในการติดตั้งสถานีเซ็นเซอร์ 15 จุดรอบพื้นที่สวนยางพารา 80 ไร่
   
2. **(ระดับประเมิน - Evaluate)**
   หากนักวิจัยต้องการส่งเสริมโครงการส่งมอบอุปกรณ์ IoT ควบคุมสระน้ำเพื่อเลี้ยงชะลอน้ำในชุมชนภาคตะวันออกที่มักประสบภัยแล้งบ่อยครั้ง จงประเมินว่าตัวเลือกโครงข่ายเชื่อมสัญญาณระหว่าง "การตั้งซิมอินเทอร์เน็ตส่วนตัวในทุกตัวบอร์ดรับส่ง" เทียบกับ "การวางเครือข่ายย่อยไร้สายภายในและพึ่งพาตัวส่งหลักตัวเดียวไปสู่บริการคลาวด์" ระบบใดมีความมั่นคงปลอดภัยและง่ายต่อการบำรุงรักษามากกว่ากันยามต้องผ่านฤดูฝนฟ้าคะนอง 10 เดือนติดต่อกัน
   
3. **(ระดับออกแบบ - Design)**
   จงออกแบบแผนผังบล็อกลอจิกซอฟต์แวร์ (Algorithmic Logic Diagram) สำหรับระบบไมโครคอนโทรลเลอร์ตรวจวัดดินทุเรียน ที่รวมตรรกะวิศวกรรมการประมวลผลสัญญาณออฟไลน์ที่จุดตรวจ, การสั่งการควบคุมวาล์วรีเลย์, การประมวลผลสถานะแบตเตอรี่, การคำนวณเข้าลูป Deep Sleep, และการดักเก็บข้อมูลกรณีสัญญาณขัดข้อง โดยเขียนอธิบายเส้นทางการเคลื่อนที่ของข้อมูลและการแผ่ลอจิกการตัดสินใจในแต่ละสถานะ
